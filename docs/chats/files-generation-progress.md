This ledger records implementation state for the
[AI image and file generation plan](./files-generation-plan.md). The image MVP
and its preview follow-up fixes have passed local validation and parallel code
and security review. Draft pull request #300 contains the implementation, and
its current GitHub Preview Build and Preview Deploy checks are green. A
read-only audit confirms the lock table is present in preview and absent in
production. The reported preview chat persisted four Google images in 9 to 12
seconds each. OpenAI smoke testing and the guarded production rollout remain.

## Current outcome

Besidka now exposes one explicit `Create image` mode for supported OpenAI and
Google chat models. The server forces a provider-neutral `generate_image`
tool, validates and persists one provider result in the existing private R2
bucket, and records the file with `source: 'assistant'` in the user's existing
file library and quota.

The chat renders honest preparing, generating, and saving states before the
authenticated image blurs into view. Generated images use compact cards,
content-sized bubbles, and an in-app fullscreen preview. The file manager uses
boxed tabs to separate `All`, `Uploaded`, and `Generated by AI` results,
prevents stale filter requests from overwriting newer results, and offers
authenticated downloads from chat, grid, and list views.

The public landing statistics now separate uploaded files from generated
images using the existing D1 provenance and MIME metadata. The home page shows
`Files uploaded` and `Images generated` tiles, while request and
scheduled-refresh wide events send privacy-safe aggregate counts to the
configured Axiom drain. Landing FAQ, comparison, feature, and README content
describe image generation without claiming that document generation already
exists.

## Preview follow-up investigation

The reported preview request completed provider work and persisted its file and
assistant message about 12 seconds after the user message. Reload displayed the
saved image, which ruled out a 20-minute provider or R2 delay.

The AI SDK Vue adapter updates streamed tool parts in place and then triggers
its shallow message reference. The generated-image child previously retained
the same tool-part object identity, so Vue kept its preparing-state computed
values until reload normalized the persisted message. The rendering boundary
now snapshots only image tool parts on the trailing assistant message whenever
the SDK triggers an update. This preserves ordinary message identity and avoids
a deep clone of chat history.

The follow-up also makes the following product corrections:

- The generic chat loader stays hidden whenever visible image progress exists.
- Generated-image skeletons and result cards use a smaller fixed width.
- Image-only assistant bubbles fit their content instead of filling the row.
- Generated and uploaded images open in an accessible in-app fullscreen modal.
- The selected-model trigger no longer repeats the active image icon.
- File source filters use the same boxed DaisyUI tabs as the main file-manager
  switcher.
- Download filenames replace stale uploaded suffixes with the authoritative D1
  MIME extension, including converted WebP uploads.
- Download headers use code-point-safe truncation, readable ASCII fallbacks,
  and bidi-control removal.
- Robots and sitemap configuration explicitly exclude private `/files/**`
  responses. Authentication remains the privacy boundary.

Deployed verification then exposed a separate shared-KV issue: active deep
research pull request #292 already used the generic `landing-stats-v3` cache
name for a different response schema. The image metrics now use
`landing-stats-image-generation-v1` plus a matching semantic client query and
scheduled invalidation key. This prevents one PR preview from reading another
feature preview's cached object. The redeployed endpoint and server-rendered
home page then returned 27 retained files split into 23 uploads and 4 generated
images at the verification timestamp.

## Second regression: hidden replies and spacer jump (2026-07-17)

The rendering-boundary fix above only snapshotted the trailing assistant
message when it carried an image tool part. Plain text and reasoning replies
still returned the same `messages` array reference, so the identity-check gap
it described was still live for ordinary chat: once a reply finished
streaming, the `data-hide-content` visibility binding could stay stale and
leave a fully rendered assistant bubble at `display:none` until an unrelated
re-render (typically submitting the next message) happened to read the live
array. This reproduced from the first assistant turn in low-render-churn
conditions and from the second turn onward in real usage.
`getRenderableChatMessages()` now returns a fresh array on every call for both
identity-passthrough branches, not only the image one.

Separately, the pending image skeleton was gated on the generic loading
indicator, which hides the instant any visible content appears - including
streamed reasoning before the tool call starts. For a model that reasons
first, this unmounted the skeleton into a genuinely empty window, shrinking
the turn and letting the browser clamp scroll position, which visibly jumped
the pinned user message until the turn finished. The skeleton is now gated on
turn status plus content that actually replaces it (a renderable tool part,
non-empty text, or a file), so reasoning alone no longer dismisses it.

This round also added a sanitized `(ref: <id>)` suffix to persisted and live
image-generation failure text, sourced from the same `providerRequestId` or
`requestId` already carried on the error payload, and logged that same
`requestId` on the failure record so a user-reported ref is directly
greppable. The `generation-busy` message no longer claims an image is
actively generating when the account-wide lock is only in its post-release
cooldown - a chat that never itself started a generation could otherwise hit
this exact message from a different chat's recent attempt and read it as a
stuck, unexplained error. The failure alert also renders full width instead
of a fixed narrow card that clipped longer messages.

### Correction: the busy error was not only a wording problem

The generation-busy wording fix above assumed the "already being generated"
error reaching a brand-new chat's first attempt was fully explained by the
per-account lock's post-release cooldown plus accurate history from an
earlier chat. Deployed-preview evidence (a fresh chat's first-ever generation
showing a busy failure and a successful image in the *same* assistant
message, with no other generation activity on the account) ruled that
explanation out and pointed at a same-turn duplicate tool call instead: some
models invoke `generate_image` more than once in one turn even though
`toolChoice` forces a single tool choice. The tool's `isExecutionClaimed`
guard (`server/utils/ai/image-generation.ts`) makes the redundant call fail
instantly with the same generation-busy classification the real lock
contention uses, so both a failed and a successful `tool-generate_image` part
can land in one message - the failure is a same-turn artifact, not a
cross-chat leak. `shouldRenderGenerateImageToolPart()` and
`normalizeGeneratedImageToolParts()` now hide a failed part, live and in
persisted history, whenever the same message has another `generate_image`
part that is not itself an error. Already-persisted messages from before
this fix keep showing the stray failure text; only new turns are covered.

*Why* the model issues the duplicate call in the first place is still open -
see "Open risks to monitor" below.

## Landing metrics and discovery content

No schema migration or second counter store is needed. The daily cached stats
query derives the metrics from canonical file records:

- `uploadedFiles` counts records with `files.source = 'upload'`.
- `generatedImages` counts records with `files.source = 'assistant'` and an
  `image/*` MIME type.
- `files` remains the backward-compatible total for API consumers.

The cache namespace and client query use a semantic image-generation version
so a stale or unrelated feature-preview response cannot omit the new fields.
This avoids collisions when multiple PR previews share a KV namespace. The
public stats route skips session
enrichment. Its request wide event and the scheduled cache-refresh event record
the total, uploaded, and generated counts plus the cache timestamp. These two
aggregate events include no prompt, filename, file ID, user ID, storage key,
generated bytes, or API key. The existing per-generation event retains normal
request correlation and records safe operational dimensions such as outcome,
provider, model, duration, MIME type, and byte count without logging prompts,
filenames, storage keys, generated content, or API keys.

The landing comparison adds image generation for every listed product. The FAQ
and feature copy explain the supported OpenAI and Google path, private file
storage, quota, preview, and download behavior. They also state that PDF,
PPTX, and XLSX generation is not implemented. Nuxt Content continues to expose
the FAQ as structured page content for search discovery.

## Implemented decisions

- Both providers use one server application tool named `generate_image`.
- The tool calls `generateImage()` with `openai.image('gpt-image-2')` or
  `google.image('gemini-3.1-flash-image')` and the current user's key.
- The supported aspect-ratio contract contains only exact shared ratios:
  `1:1`, `2:3`, and `3:2`.
- Image mode takes precedence when a crafted request also selects web search.
  The server omits the provider web tool and exposes only `generate_image`.
- The route rejects unsupported model and tool combinations before it writes
  the user message or calls a provider.
- The tool claims execution synchronously before its first `await`, while a
  token-owned D1 lease allows only one image generation per user. The lease
  expires after 10 minutes for crash recovery and becomes a 10-second cooldown
  after the current owner releases it.
- The tool persists through `persistFile()` with assistant provenance and
  returns compact metadata. Generated bytes never cross the chat stream.
- Generated images use the existing private bucket. The `source` field, not an
  R2 prefix or second bucket, separates generated and uploaded files.
- The server derives safe filenames and authenticated view/download URLs from
  validated storage keys. It ignores URLs supplied in tool output.
- Assistant normalization re-reads the file by ID and owner from D1, then
  verifies its storage key, name, size, MIME type, assistant source, and
  provider provenance before deriving the persisted file URL.
- Strict request schemas accept only current user text and file parts. Forged
  assistant or tool parts cannot enter chat persistence through the POST body.
- The client renders generated-image tools only on assistant messages and
  validates the complete ready payload before deriving local URLs.
- Successful generation resynchronizes an active `showFiles` share. Branching
  strips tool parts so copied shared content cannot become executable UI.
- Assistant messages persist `image_generation` as non-executable tool-use
  metadata. Shared views expose that metadata only when `showMetadata` is on,
  always strip executable tool parts, and never copy the metadata into a chat
  created from a shared branch.
- Live tool failures and reload persistence use explicit error allowlists.
  Unknown or malformed provider text degrades to generic safe guidance.
- Neither provider exposes trustworthy incremental progress through this path.
  The UI therefore shows named phases, a stable skeleton, and a final blur-in
  without a numeric percentage.
- OpenAI hosted partial snapshots remain deferred because they add cost and
  would move base64 through the chat stream.
- Image calls use no automatic retry after an ambiguous provider request.

## Purpose-built image model catalog

The provider pickers now append current, non-deprecated image-generation
models after the existing general-purpose models:

- OpenAI: GPT Image 2.
- Google: Gemini 3.1 Flash Image, Gemini 3.1 Flash Lite Image, Gemini 3 Pro
  Image, and Gemini 2.5 Flash Image.

Deprecated GPT Image aliases, deprecated Imagen 4 models, and all video models
are intentionally excluded. Purpose-built image models declare `tools: []`, so
the model menu does not show reasoning or web-search badges for them, but it
still shows the image-generation badge — the picker checks the model's
`imageGeneration` capability, not just its `tools` array, so the badge that
already existed for general-purpose models also appears here. That capability
automatically enables the existing image-generation tool and prevents it from
being disabled while the model is selected; the "Create image" toggle renders
in its normal active state rather than the browser's disabled/greyed style, so
a required-on model still visually reads as on.

The server never sends an image-only model to `streamText`. It uses a small
same-provider controller model to make the forced tool call, then passes the
selected image model to the AI SDK image API. Provider provenance records the
selected image model. Pricing labels reflect the standard output size and
quality currently requested by Besidka and note that provider input charges
are additional. OpenAI does not publish a token context window for GPT Image 2,
so its registry context and output-token values are zero rather than invented.

An independent code review and security audit ran against this commit before
merge. Both provider utilities now fail closed (400) instead of silently
falling back to the raw model string when `getModel()` cannot resolve a
catalog entry, and the chat input's `toggleWebSearch` guard now mirrors
`toggleImageGeneration`'s required-model guard for defense in depth. Neither
gap was reachable through any current call site; both are hardening only.

## Open risks to monitor

These risks require evidence during the manual preview rollout:

- OpenAI organization verification and model access vary by user account.
- Provider model lifecycle and supported chat-model allowlists can change.
- A provider can finish after an edge timeout, so a later manual retry can
  still create another provider charge after the 10-minute crash lease expires.
- The SDK buffers the final result before Besidka can reject an image above the
  10 MiB persistence bound.
- Uploads can race an image generation near the storage limit. The final quota
  check compensates the losing persistence attempt.
- Google safety and no-image finish reasons can differ across model releases.
- Existing text token cost cannot represent total image-generation cost.
- OpenAI hosted partial images remain deferred because they add output cost and
  move base64 through the chat stream.
- Open pull request #292 also changes landing statistics and content. Whichever
  feature merges second must preserve both its `researches` field and this
  branch's `uploadedFiles` and `generatedImages` fields, then choose a new
  combined semantic cache name and client query.
- Root cause unconfirmed: why a model calls `generate_image` more than once
  in one turn under a forced `toolChoice`. The rendering/persistence fix
  hides the resulting redundant-call artifact, but each duplicate call still
  spends provider quota and a real generation attempt on the doomed call
  before the guard rejects it. Confirmed only via deployed-preview evidence
  (Google, per the "Google AI" provenance on the successful image) - not yet
  reproduced against OpenAI, and not reproducible locally without live
  provider credentials, since the mock `/chats/test` harness scripts a fixed
  chunk sequence rather than exercising real model tool-calling behavior.
  Investigate whether the Google provider's forced function-calling mode
  (`functionCallingConfig: { mode: 'ANY' }`) allows more than one call to an
  `allowedFunctionNames` entry per turn, and whether AI SDK v7 exposes a way
  to cap tool calls per step for this request.
